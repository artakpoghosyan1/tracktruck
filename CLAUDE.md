## Product Overview

TrackTruck is a **GPS route-simulation and live-tracking platform**. An operator designs a route on a map, the server simulates a vehicle traveling along that route in real time (respecting per-segment speed limits or a target arrival time), and a shareable public page renders the moving position live over WebSocket. The position shown on the public page is **computed by the simulation engine** — it is not data from a physical GPS device.

It has two distinct surfaces:

_**Admin dashboard** — authenticated. Customers build routes on an interactive Mapbox map: set start/end points, draw the path, add intermediate stops with dwell times, and add steering waypoints. They then *activate* a route (which mints a public share link) and *start* it (which begins the simulation). While a route runs they can pause, resume, change the speed/target duration, add or remove stops, and re-route via waypoints.

**Public tracking map** — a zero-auth page reached through a share link. It shows the simulated vehicle's live position, bearing, the route polyline, and stops, updating in real time via WebSocket. When the route finishes it renders a completed state; when a link is inactive/unknown it renders a "no active route" state.

> Note on framing: this is a *simulation* tool. The moving marker reflects a synthetic position generated by the engine. Appropriate uses are things like demos, QA/testing of tracking UIs, training, and route/logistics visualization where viewers understand the position is simulated.

### Organizations & onboarding_

Customers are modeled as **organizations**. Onboarding flow:

1. A **super_admin** (platform owner) creates an organization with a name, a route quota (`organizations.routeLimit` — the org's "tier"), and an `isPaid` flag.
2. A platform **admin** or **super_admin** authorizes the customer's email in `allowed_emails`, assigns it the `org_admin` role, and links it to the organization. That person becomes the organization's administrator.
3. The **org_admin** signs up/logs in, then adds their own users (also into `allowed_emails`, role `user`, linked to the same org) and allocates each user a slice of the org's route quota.

The org's total quota is a pool. Each member is given an individual `routeLimit` carved out of that pool; the sum of all members' allocations cannot exceed the org's `routeLimit` (`remaining = org.routeLimit − Σ members' routeLimit`). See `artifacts/api-server/src/routes/org.ts`.

### User roles

There are **four** roles (`super_admin`, `admin`, `org_admin`, `user`). The role lives on both `users.role` and `allowed_emails.role`; on login the value from `allowed_emails` is synced onto the user.

| Role | Who | What they can do |
|---|---|---|
| `user` | Org members | Create/manage their own routes; activate up to their personal allocation. **Subject to quota.** |
| `org_admin` | A customer's org administrator | All of `user` + manage users within their own organization and distribute the org's route pool among them (`/org/*`). **Subject to quota** (their own activations count too). |
| `admin` | Platform staff | Manage `allowed_emails` and `organizations` across the platform: authorize new emails (as `user`/`org_admin` only), create/edit orgs, set `isPaid` and `routeLimit`. **Exempt from quota.** Cannot create `admin`/`super_admin`. |
| `super_admin` | Platform owner (us) | Everything, including changing any user's role and onboarding/deactivating orgs. **Exempt from quota.** |

Middleware: `requireAuth`, `requireOrgAdmin`, `requireAdmin`, `requireSuperAdmin` in `artifacts/api-server/src/middlewares/auth.ts`. Each tier also satisfies the checks below it (e.g. a `super_admin` passes `requireOrgAdmin`).

### Access control (the `allowed_emails` gate)

**Nobody can sign up or log in unless their email is pre-added to `allowed_emails`.** On signup/login the server looks up the email there to resolve its `role`, `organizationId`, and quota. If the record's `isPaid` is false the server returns **402**. If the email isn't present at all it returns **403**.

The `ROOT_ADMIN_EMAIL` env var designates one email that always resolves to `super_admin` and **bypasses the `allowed_emails` gate entirely** (no row required).

### Route quota model

Quota is enforced **per user**, drawn from the org pool:

- `allowed_emails.routeLimit` — how many routes this member may ever activate (their allocation).
- `allowed_emails.usedRoutes` — incremented by 1 **each time the member activates a route**. **It is never decremented** — deactivating, pausing, completing, or deleting a route does not give the slot back. So `usedRoutes` is a lifetime activation count, and activation is blocked once `usedRoutes >= routeLimit` (HTTP **403** `quota_exceeded`).
- The check applies only to `user` and `org_admin` roles; `admin`/`super_admin` are exempt.
- If the member's org is `isPaid = false`, activation returns **402** `payment_required`.

Enforcement: `POST /routes/:id/activate` in `artifacts/api-server/src/routes/simulation.ts`.

### Payment model

No payment processor is integrated. Billing is handled out-of-band — customers pay the platform operators directly for their subscription/tier. Inside the app, payment status is a manual CRM-style control: a `super_admin`/`admin` flips `isPaid` (on the organization and/or the `allowed_emails` record) and sets the org's `routeLimit` (tier). Flipping `isPaid` off blocks signup/login (402) and route activation (402) for that org's members.

### Route building & in-flight controls

A route is start point + end point + a **polyline** (array of `[lng, lat]`) plus optional stops and waypoints.

- **Stops** (`route_stops`) — named intermediate points with a **dwell time** (`durationMinutes`, default 5). While the vehicle is parked at a stop the engine broadcasts a countdown (`stopDwellRemainingS`) and the public status becomes `at_stop`. Stops can be added/removed while the route is `in_progress`; the engine recomputes from the current position so the marker doesn't jump backward.
- **Waypoints** (`routes.waypoints`, also baked into the polyline) — steering points that shape the path but have no dwell. Editing the polyline/waypoints on an `in_progress` route keeps the vehicle moving from its current position onto the new path (it only fully resets if the start/end points change).
- **Target duration / ETA** — instead of pure speed-limit timing, a user can pin how long the trip should take. `routes.customDurationS` + `customDurationEnabled` make the engine apply a speed multiplier (`estimatedDurationS / customDurationS`) so the vehicle arrives at the chosen time. Speed/duration changes are **unlimited** and do **not** count as route edits.
- **One edit while running** — for `user` role only, a started route (`in_progress`/`paused`) may have its *meaningful* fields (start/end/name/route geometry) changed **only once**; further changes return **403**. Tracked via `routes.updateCount`. Speed-only and waypoint-only changes don't increment it. `org_admin` and above are not limited. (The `route_revisions` table exists but is **not used** to enforce or record this.)
- **Pause/resume** — pausing preserves accumulated elapsed time (`effectiveElapsedMs`) and broadcasts a speed-0 snapshot so the public map stops moving; resuming continues from where it left off.

### Route lifecycle

`draft → ready → in_progress ↔ paused → completed`

- **Activate** (`draft → ready`): validates the polyline has ≥2 points, checks quota/payment, increments `usedRoutes`, and **mints a share link** — a random hex `token` with a 7-day `expiresAt`. ⚠️ The `expiresAt` field is currently **set but not enforced**; the public page is gated solely by the link's `active` flag.
- **Start** (`ready → in_progress`): records `startedAt` and the simulation begins ticking.
- **Pause/Resume** (`in_progress ↔ paused`).
- **Complete**: when the vehicle reaches the end of the polyline the engine waits a 5-second grace period, then marks the route `completed`. **It does not deactivate the share link** — the public page just shows a completed state. Share links are set `active = false` on route **deletion**, when a route is **re-activated** (old links cleared before minting a new one), and during **startup cleanup**.
- An `expired` status value exists in the schema but is **never auto-set** by the engine.

## Architecture

pnpm monorepo. Two artifacts consume shared libs:

```
artifacts/api-server    Express 5 API (auth, routes, simulation, org, admin, public, WebSocket)
artifacts/tracktruck    React + Vite frontend (admin dashboard, route builder, public tracking)

lib/db                  Drizzle ORM schema + PostgreSQL connection (@workspace/db)
lib/api-spec            OpenAPI 3.1 spec + Orval codegen config
lib/api-zod             Generated Zod request/response schemas (from OpenAPI via Orval)
lib/api-client-react    Generated TanStack Query hooks (from OpenAPI via Orval)
```

API route modules live in `artifacts/api-server/src/routes/`: `auth`, `routes` (CRUD + stops + speed/duration), `simulation` (activate/start/pause/resume), `org` (org-admin user/quota management), `admin` (platform allowed-emails + org management), `public` (zero-auth tracking), `ws` (WebSocket), `health`.

**Never hand-edit** `lib/api-zod/src/generated/` or `lib/api-client-react/src/generated/` — these are produced by `pnpm --filter @workspace/api-spec run codegen` from `lib/api-spec/openapi.yaml`.

## API / Codegen Flow

`lib/api-spec/openapi.yaml` → Orval → two outputs in parallel:
- `lib/api-zod/src/generated/` — Zod schemas used by the server for request validation
- `lib/api-client-react/src/generated/` — React Query hooks used by the frontend

When adding or changing an API endpoint, update `openapi.yaml` first, run codegen, then implement the route and consume the hook.

## Database

Drizzle ORM with PostgreSQL. Schema files live in `lib/db/src/schema/`, re-exported from `index.ts`. The `db` export and all table references come from `@workspace/db`.

Key tables:
- `users`, `oauth_accounts`, `refresh_tokens` — identity & sessions.
- `allowed_emails` — the access gate; holds `role`, `organizationId`, `isPaid`, `routeLimit`, `usedRoutes`.
- `organizations` — the customer/tenant; holds `name`, `isPaid`, `routeLimit` (org pool/tier).
- `routes`, `route_stops` — route definition, geometry, stops & dwell times.
- `simulation_states` — live run state per route (`startedAt`, `pausedAt`, `effectiveElapsedMs`, progress, current-stop tracking).
- `share_links` — public tracking tokens (`token`, `active`, `expiresAt`).
- `live_snapshots` — last-known position snapshot per route.
- `route_revisions` — **defined but currently unused** in the API.

Route lifecycle statuses: `draft → ready → in_progress ↔ paused → completed` (plus a defined-but-unused `expired`).

Schema changes: edit the schema file, then run `pnpm --filter @workspace/db run push`. No migration files are used; schema is pushed directly.

## Auth

- Email/password signup requires the email to be pre-authorized in `allowed_emails` by an admin (403 otherwise; 402 if unpaid).
- Google OAuth also requires the email to be in `allowed_emails` (except `ROOT_ADMIN_EMAIL`).
- JWT access + refresh token pair. Access token stored in `localStorage` as `tracktruck_token`.
- On every login/`/auth/me`, the user's `role` and `organizationId` are re-synced from `allowed_emails`.
- Frontend global fetch interceptor (`artifacts/tracktruck/src/lib/api-interceptor.ts`) injects the Bearer token on all `/api` requests and redirects to `/login` on 401.
- `ROOT_ADMIN_EMAIL` env var bypasses `allowed_emails` and always resolves to `super_admin`.

## Simulation Engine

The main server calls `startSimulationEngine()` in `artifacts/api-server/src/lib/simulation-engine.ts`, which spawns a **Node worker thread**. The actual tick loop runs **inside that worker** in `artifacts/api-server/src/lib/simulation-worker.ts` (in dev it's loaded via `simulation-worker-bootstrap.mjs` + tsx; in prod via the compiled `.cjs`). The worker posts position snapshots back to the main thread, which fans them out over WebSocket.

Each tick (`TICK_INTERVAL_MS = 2000`, every 2 s):
1. Queries all `in_progress` routes + their `simulation_states` (route definitions are cached per route).
2. Computes elapsed time (pause-adjusted, scaled by the custom-duration multiplier) and the truck position via `positionAlongPolyline()` in `src/lib/geo.ts` (Haversine cumulative distance → segment interpolation → bearing).
3. Broadcasts a snapshot (`lat`, `lng`, `bearing`, `speedMph`, `progressPercent`, stop info, `status`) to that route's share token(s) and admin subscribers.
4. Persists position to the DB periodically (`DB_SAVE_INTERVAL_TICKS = 5`, ~every 10 s) rather than every tick.
5. On reaching the end, holds a `COMPLETION_GRACE_S = 5` grace period, then marks the route `completed`.

WebSocket channels (`artifacts/api-server/src/routes/ws.ts`): `/api/public/ws/track/:token` (unauthenticated, keyed by share token) and `/api/admin/ws/routes/:routeId?token=<jwt>` (JWT in query param, keyed by route id).
